# People Dossier

The primary view of Mental Metal V2. For each person, synthesise everything the AI knows from across ALL meetings — not just 1:1s, but every conversation where they were mentioned or participated. Surface contradictions, performance signals, open commitments, and pre-meeting prep.

## What the Dossier Shows

### AI-Generated Synthesis
A narrative paragraph (2-5 sentences) that answers: "What should I know about this person right now?" Generated by feeding all recent captures mentioning this person to the AI.

The synthesis should:
- Summarise recent context (what projects they're on, what came up recently)
- Highlight any contradictions across sources (different people saying different things)
- Note positive and negative signals (without judgment — just surfacing what was said)
- Flag anything that looks time-sensitive or unresolved

### Open Commitments
- Commitments where this person is the `PersonId` (either direction)
- Grouped by: mine-to-them, theirs-to-me
- Sorted by: overdue first, then due date
- Status shown: open, overdue

### Transcript Mentions
- Chronological list of all Captures where this person is linked (via extraction)
- Each entry shows: date, meeting title, brief context snippet (the extraction's `Context` field for this person's mention)
- Click to view full capture

### Unresolved Mentions
- Names from extractions that might be this person but weren't auto-resolved
- Action: "Yes, this is [person]" → adds alias, re-links capture

## Generation Process

### On-Demand Synthesis
The dossier is generated when the user opens the person detail page:

1. Query all Captures linked to this PersonId, ordered by date descending
2. Take the most recent N captures (default: 20, configurable) — don't send entire history
3. For each capture, include: date, title, extraction summary, and the specific context where this person was mentioned
4. Send to AI with prompt: generate a synthesis of this person's current situation
5. Cache the result with a TTL (default: 1 hour or until a new capture links to this person)

### Pre-Meeting Prep Mode
Accessed via: `GET /api/people/{id}/dossier?mode=prep`

Same generation process but with a different AI prompt tuned for meeting preparation:
- "You're about to meet with this person. Here's what you should know."
- Emphasise: open commitments (both directions), unresolved items, recent context changes
- De-emphasise: historical patterns, signals from weeks ago

### Cache Invalidation
- Cache key: `dossier:{personId}:{mode}`
- Invalidated when: new capture linked to this person (via extraction pipeline)
- On cache miss: generate fresh (may take 2-5 seconds for AI call)
- On cache hit: return immediately, show "last updated X ago" indicator

## API

### New Endpoints
- `GET /api/people/{id}/dossier` — returns the full dossier (synthesis + commitments + mentions)
  - Query params: `mode=default|prep`, `captureLimit=20`
  - Response includes: `synthesis` (AI text), `commitments` (list), `mentions` (list with context), `unresolvedMentions` (list), `generatedAt` (timestamp)
- `POST /api/people/{id}/dossier/refresh` — force regeneration (bypass cache)

### Existing Endpoints Used
- `GET /api/commitments?personId={id}&status=open` — open commitments for this person
- `GET /api/captures?personId={id}` — captures linked to this person

### Person List Page
- Each person row shows: name, type, mention count (last 7 days), open commitment count
- Click navigates to dossier view
- Sort options: most mentions, most recent mention, alphabetical

### Pre-Meeting Prep
- Accessible from dossier page via "Prep for Meeting" button
- Switches AI prompt to prep mode
- Same layout but synthesis is tuned for meeting readiness
- Could also be triggered from a calendar integration in the future (non-goal for V2)

## Performance Considerations

- Dossier generation involves an AI call — expect 2-5 second latency on cache miss
- Show loading skeleton while generating
- Cache aggressively (1 hour TTL)
- Invalidate only when new relevant capture is processed
- Limit captures sent to AI (default 20 most recent) to control token usage and latency
- Consider background pre-generation for people with high mention frequency (future optimisation)
